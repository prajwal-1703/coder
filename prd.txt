# Why vLLM instead of llama.cpp?

Both are excellent, but they target slightly different priorities.

### llama.cpp

Best for:

* CPUs
* GGUF models
* Lightweight deployments
* Lower-end hardware

### vLLM

Best for:

* High-end NVIDIA GPUs
* Multiple concurrent users
* High throughput
* Hugging Face models
* Long-running inference servers

An RTX A6000 is exactly the kind of hardware vLLM is designed to use well.

# One architectural improvement I'd make from the start

The only thing I'd add beyond your current requirements is a **local embedding model** (for example, a lightweight sentence embedding model served alongside vLLM).

Why?

Modern coding assistants aren't just generating code—they also search your repository, retrieve relevant files, and provide context-aware answers. That retrieval step is powered by embeddings, not the main LLM.

So I'd plan for two model categories from day one:

* **Inference model:** Qwen3-Coder (code generation and reasoning)
* **Embedding model:** for semantic code search and repository indexing

Even if you don't enable repository indexing immediately, leaving room for an embedding service in your Docker Compose layout will make it much easier to evolve the platform into a genuinely powerful local coding assistant later.
